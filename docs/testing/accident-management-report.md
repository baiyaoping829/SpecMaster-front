# 事故案例模块｜事故管理子模块｜测试报告（E2E + 人工验收口径）

## 结论

存在阻塞性缺陷：需求范围中多项关键能力未实现或不可验证（富文本、地理位置、自动评级/审核流、整改跟踪、统计导出、权限与审计日志、敏感字段脱敏）。附件上传端到端链路已补齐并通过自动化回归。

## 测试环境

- 前端：`http://localhost:5173/`
- Node API：`http://localhost:3001/`
- 事故案例 FastAPI：`http://localhost:8001/`
- 浏览器自动化：Playwright（Chromium、Firefox）

## 已执行项（通过）

### 1) UI 端到端（Playwright 回放）

- 覆盖：必填校验、事故新增/编辑/删除、附件上传端到端（FastAPI→MinIO→attachment_keys）
- 结果：Chromium/Firefox 均通过
- 脚本：
  - `e2e-accident/accident-management.spec.ts`
  - `playwright.accident.config.ts`

### 2) 边界数据（≥200）

- 通过 `backend/scripts/seed_accidents.py` 写入 220 条事故案例（标题超长截断、特殊字符、随机字段）
- 结果：成功创建 220 条

### 3) 接口性能与并发（冒烟）

- 50 并发、持续 60 秒、循环访问 `GET /api/v1/accidents?page=1&size=20`
- 结果（本机测得）：
  - avg 133.8ms，p95 225ms，p99 287ms
  - 4xx/5xx：0
- 并发提交冲突验证：
  - 50 并发对同一记录同版本 PUT 更新：200=1，409=49（符合乐观锁预期）
- 脚本：
  - `tools/perf/accidents-load.mjs`
  - `tools/perf/accidents-concurrency.mjs`

说明：未执行 30 分钟长稳压测与“内存泄漏 <50MB”验证（会长时间占用会话/需独立机器与监控）。脚本已支持参数化，按文档可一键拉满。

## 缺陷清单（阻塞/高优先）

### AC-BLOCK-002｜富文本描述未实现（阻塞）

- 期望：事故描述/经过支持富文本编辑与安全渲染
- 实际：使用普通 textarea/文本输入，且未定义富文本存储/渲染策略

### AC-BLOCK-003｜地理位置选择未实现（阻塞）

- 期望：地图选点/行政区选择，落库并可检索/统计
- 实际：无地理位置组件与数据字段

### AC-BLOCK-004｜事故分级与分类自动评级/审核流未实现（阻塞）

- 期望：自动评级算法、人工调整、多级审核流（状态机、审计留痕）
- 实际：仅静态字段录入，未见算法/工作流/审批记录

### AC-BLOCK-005｜整改措施跟踪未实现（阻塞）

- 期望：措施拆解、责任人、时限、状态流转、催办与统计
- 实际：无相关数据模型/API/UI

### AC-BLOCK-006｜统计与导出未实现（阻塞）

- 期望：多维筛选统计、图表可视化、导出 PDF/Excel
- 实际：无统计页/导出接口

### AC-BLOCK-007｜权限与日志/脱敏未实现（阻塞）

- 期望：角色权限矩阵、操作审计日志、敏感字段脱敏
- 实际：事故案例 FastAPI 当前未接入鉴权与审计

## 建议的下一步（按优先级）

1. 定义事故分级/审核流/整改跟踪的最小闭环（状态机 + 审批/整改记录表）
2. 补齐统计与导出（筛选、图表、PDF/Excel 导出接口）
3. 引入权限与审计：至少做到“读写鉴权 + 操作留痕 + 脱敏策略”

# 架构设计（多数据库 + Redis + MinIO + 可观测性）

## 1. 目标与边界

本仓库提供一套可运行的“统一数据访问架构骨架”，支持：

- 多数据库：PostgreSQL / MySQL / SQLite（同一服务实例运行时可切换）
- 事务：写操作强制事务；支持事务回滚验证；默认 READ COMMITTED
- 并发控制：乐观锁（version）与悲观锁（预留接口与策略）
- 缓存：Redis Cache-Aside + 分布式锁防击穿 + TTL + 主动失效
- 最终一致性：DB Outbox + Redis Streams 事件驱动的缓存失效
- 对象存储：MinIO（S3 兼容）+ 以 SHA-256 为对象键 + 预签名 URL
- 可观测性：Prometheus 指标、OpenTelemetry 链路追踪

未在代码层面“自动落地”的部分（需要基础设施能力支撑）以运维方案与配置模板形式提供：DB 主从/集群与自动故障转移、MinIO 多站点复制/版本控制、KMS 等。

## 3. 多数据库与动态切换

### 3.1 数据源配置

- 默认数据源由 `DB_DIALECT` 决定：`sqlite | postgres | mysql`
- PostgreSQL/MySQL 连接由 `PG_*`、`MYSQL_*` 环境变量提供
- SQLite 文件由 `SQLITE_FILE` 指定

### 3.2 运行时动态切换

- 每个请求可通过 Header `X-DB-DIALECT: postgres|mysql|sqlite` 选择数据源
- 管理接口支持切换默认数据源：
  - `POST /api/admin/datasource/switch`（需管理员 JWT）

## 4. 事务与并发控制

- 写操作：业务服务在写路径统一走 `withTransaction()`，确保 ACID（由底层数据库保证）
- 隔离级别：默认 READ COMMITTED；可通过 `DB_READ_ISOLATION` 调整
- 乐观锁：表内 `version` 字段，更新时 `where id=? and version=?`，成功后 version+1
- 悲观锁：PostgreSQL/MySQL 可用 `SELECT ... FOR UPDATE`（建议在后续扩展的领域服务中按需要使用）

## 5. 缓存与最终一致性

- Cache-Aside：读缓存未命中 → 查库 → 回写；写落库后删除缓存
- 防击穿：Redis 分布式锁（Redlock）对热点 key 做互斥加载
- TTL：实体级 key 支持 TTL；建议生产采用 TTL + 主动失效结合
- 最终一致性：
  - Outbox：事务内写 outbox_events
  - Publisher：轮询 pending outbox → 写入 Redis Streams（`specmaster:events`）
  - Consumer：消费失效事件 → 删除缓存 key

一致性校验脚本见：[cache-consistency-check.mjs](file:///h:/learnproject/04%20solo_builder/server/scripts/cache-consistency-check.mjs)

## 6. 对象存储（MinIO）

- 上传：服务端接收 multipart 文件 → 落临时文件 → 计算 SHA-256 → 以 sha256(+ext) 为对象键上传 MinIO
- 元数据：存入 `file_objects` 表，可分页检索
- 访问：默认私有桶，通过预签名 URL 限时访问

## 7. 可观测性

- 指标：`GET /metrics`（Prometheus 抓取）
- 追踪：OpenTelemetry SDK（OTLP HTTP 导出，`OTEL_EXPORTER_OTLP_ENDPOINT`）
